### 01. 작업 의뢰 시 배경·의도 파악 누락 및 확인 미흡

- **발생 상황**: 작업 의뢰를 받은 후, 의뢰자의 숨겨진 배경이나 정확한 요구 맥락을 깊이 파악하지 않은 상태에서 직관에 의존해 작업에 조기 착수함.
- **지적 내용**: 의뢰자가 기대한 결과물의 방향성이나 비즈니스적 맥락과 싱크가 맞지 않음. 모호한 부분이 있음에도 자의적으로 해석하여 진행한 점을 지적받음.
- **원인 분석**:
  - 일정을 맞추려는 조급함으로 인해 요구사항 정의(요건 정의) 단계를 느슨하게 통과함.
  - 질문을 많이 하는 것이 업무 진행을 더디게 하거나 프로답지 못하게 보일까 염려하여 혼자 판단함.
- **재발 방지 대책 (Action Item)**:
  - [ ] **[착수 전 필수 3대 체크]**: 모든 작업 의뢰는 `1) 의뢰 배경`, `2) 비즈니스 맥락`, `3) 최종 의도(목적)`를 완벽히 이해하기 전까지 개발(코드 작성/문서화) 프로세스를 시작하지 않는다.
  - [ ] **[50% 룰 적용]**: 요구사항이 100% 명확하지 않고 애매한 영역이 단 1%라도 있다면, 즉시 의뢰자에게 슬랙/메일 또는 구두로 확인용 질문(Q&A)을 던져 싱크를 맞춘다.
  - [ ] **[역제안 가이드]**: "제가 이해한 맥락은 A이며, 이를 위해 B 방향으로 구현하려 하는데 의도하신 바가 맞습니까?" 형태로 질문 구조를 공식화한다.

### 02. 팀 공지사항(수정/추가) 리마인드 및 준수 미흡

- **발생 상황**: 프로젝트 진행 중 슬랙(Slack), 노션(Notion), 위키 등에 공지사항이 새로 업데이트되거나 수정되었으나, 기존 작업 방식대로 관성적으로 진행하여 변경된 룰을 일시적으로 놓침.
- **주의 필요성**: 프리랜서는 현장의 최신 프로세스 변화에 가장 민각하게 반응해야 함. 공지 준수 누락은 단순 실수를 넘어 팀의 협업 신뢰도를 떨어뜨리는 요인이 됨.
- **원인 분석**:
  - 내 작업(코딩/문서 작성)에 몰두하다 보니 공지 채널의 알림이나 업데이트 피드를 실시간으로 확인하지 못함.
  - '기존에 하던 방식이 맞겠지'라는 자의적 판단과 관성으로 확인을 소홀히 함.
- **재발 방지 대책 (Action Item)**:
  - [ ] **[루틴화]**: 매일 업무 시작 직후(오전)와 업무 종료 직전(오후), 최소 하루 2회 이상은 팀의 전체 공지 채널 및 위키 히스토리를 의도적으로 체크한다.
  - [ ] **[북마크 활용]**: 수정되거나 추가된 중요 공지사항은 즉시 나만의 북마크나 메모에 박제하고, 당일 작업 내용에 영향이 있는지 최우선으로 검토한다.
  - [ ] **[의문 시 재확인]**: 공지 내용이 기존 작업 방식과 충돌하거나 애매한 부분이 있다면, 혼자 판단하지 않고 즉시 담당자에게 변경 사항의 적용 범위를 확인한다.

### 03. 보고용 자료 내 모호한/애매한 표현 사용 지양

- **발생 상황**: 진척 상황이나 트러블슈팅 결과를 공유하는 보고용 문서(또는 슬랙 보고)에서 명확하지 않은 추측성 단어나 정성적인 표현을 사용함.
- **지적 내용**: 보고를 받는 의뢰자나 관리자 입장에서 정확한 상태 파악이 불가능하고, 판단을 흐리게 만든다는 지적을 받음.
- **원인 분석**:
  - 정확한 데이터나 수치를 확인하는 데 시간이 걸리거나 귀찮아서 직관적인 느낌으로 기술함.
  - 확정되지 않은 상황에 대한 책임을 회피하기 위해 무의식적으로 방어적인 표현을 선택함.
- **재발 방지 대책 (Action Item)**:
  - [ ] **[금지어 설정]**: 보고서 작성 시 `'~인 것 같다'`, `'대략'`, `'거의 완료'`, `'어느 정도'` 같은 추측성 및 정성적 단어는 절대 사용하지 않는다.
  - [ ] **[수치화·정량화]**: 모든 진척도와 이슈는 숫자로 말한다. (예: "거의 다 됐습니다" ❌ -> "전체 10개 태스크 중 8개 완료하여 현재 진척률 80%입니다" ⭕)
  - [ ] **[사실과 의견 분리]**: 확정되지 않은 내용을 보고할 때는 [사실(Fact)]과 [내 의견/추측(Opinion)]을 명확히 칸을 나누어 서술한다.
