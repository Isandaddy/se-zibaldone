# 🔍 AI 코드 리뷰 및 프로세스 체크리스트

> AI가 임의로 코드를 생성하여 발생할 수 있는 리워크(Rework)를 방지하고, 프로젝트의 요건과 실장 룰에 완벽히 부합하는 코드만 생산하도록 제어하는 가이드라인입니다.

---

## 🛠️ 1. 개발 착수 전 필수 확인 프로세스 (Pre-Generation Verification)

### 📌 규칙 01. 코드 생성 전 요건 및 실장 룰 사전 확인 의무화

- **적용 가이드**: AI는 어떠한 소스코드(FE/BE/Test)도 임의로 즉시 생성해서는 안 되며, 반드시 구현 전에 개발자와 먼저 요건 및 규칙에 대해 싱크를 맞춰야 합니다.
- **AI 필수 준수 단계**:
  1.  **[1단계: 요구사항 재정의]** 개발자가 요청한 작업의 요건과 구현 목적을 AI가 이해한 대로 요약하여 설명합니다.
  2.  **[2단계: 관련 규칙 검토]** 본 프로젝트의 다른 실장 룰 파일(`01-security-and-conflict.md`, `02-backend-architecture.md` 등) 중 이번 구현에 적용해야 하는 보안 및 아키텍처 규칙이 무엇인지 스스로 리스트업합니다.
  3.  **[3단계: 구현 설계 및 승인 요청]** 구체적인 코드 생성에 들어가기 전, 어떤 파일 구조로 어떤 로직을 짤 것인지 대략적인 '구현 설계 의도'를 먼저 제안하고 개발자의 확인(승인)을 받습니다.
- **실장 방식**: 개발자가 구현 설계를 확인하고 "확인했으니 코드를 생성해 줘"라고 최종 승인(Go) 사인을 주기 전까지는 실제 소스코드 출력을 대기합니다.

### 📌 규칙 02. 기존 파일 수정 시 소스코드 사전 확인 의무화 (No Assumptions)

- **적용 가이드**: AI가 기존에 존재하는 파일을 수정하거나 코드를 추가할 때, 기억이나 추측에 의존해 코드를 생성하는 것을 전면 금지합니다.
- **AI 필수 준수 사항**:
  - 작업을 수행하기 전, 수정 대상이 되는 파일의 최신 전체 내용(Full Context)을 반드시 먼저 읽고 분석(Read/Load)해야 합니다.
  - 파일 내부의 기존 내보내기(Exports), 상태 관리(State), 의존성 패키지(Dependencies) 구조를 완벽히 파악한 후에만 수정 코드를 작성합니다.
- **실장 방식**: 파일의 일부분만 보고 "아마 이런 구조일 것이다"라고 자의적으로 추측하여 코드를 짜는 행위를 금지하며, 반드시 최신 소스코드와 완벽히 동기화된 상태에서 변경 사항(Diff)을 생성합니다.

### 📌 규칙 03. 예외 상황 및 타입 불일치 발견 시 작업 중단 및 확인 요청

- **적용 가이드**: 개발 진행 중 요건의 모순, 로직상 확인이 필요한 애매한 점, 또는 프론트엔드-백엔드-DB 간의 타입 불일치(Type Mismatch)가 감지되면 AI는 즉시 코드 생성을 중단해야 합니다.
- **보안 및 실장 금지 사항**:
  - 불일치를 무시하고 임의로 타입을 `any`로 우회하거나 추측성 예외 처리 코드를 삽입하는 행위를 전면 금지합니다.
- **AI 필수 준수 사항**:
  - 문제를 발견한 즉시 개발자(사용자)에게 상황을 보고합니다.
  - 보고 시에는 **1) 불일치가 발생한 위치와 내용**, **2) 예상되는 사이드 이펙트**, **3) 가능한 해결 방안(Option)**을 간결하게 정리하여 제시합니다.
- **실장 방식**: 개발자가 해당 문제를 검토하고 인프라 프로퍼나 팀의 담당자에게 확인하여 명확한 가이드를 제공할 때까지 해당 모듈의 구현을 대기합니다.

### 📌 규칙 03. 수정 대상 연관 소스코드 및 파일의 영향 범위 분석 필수 (Impact Analysis)

- **적용 가이드**: AI가 특정 파일이나 함수를 수정할 때, 오직 해당 수정 대상(Target)만을 확인하고 코드를 변경하는 행위를 전면 금지합니다. 수정으로 인해 예기치 못한 버그(Side Effect)가 발생하지 않도록, 반드시 해당 코드를 참조하거나 의존하고 있는 **연관 파일 및 주변 코드의 영향 범위를 선행 조사**해야 합니다.
- **AI 필수 준수 영향 범위 체크리스트**:
  - **[내부 의존성]**: 동일 파일 내에서 수정 대상 함수/변수를 참조하고 있는 다른 로직이 있는지 전수 확인합니다.
  - **[외부 레이어]**: 수정 대상이 공개(Export) 함수인 경우, 상위 레이어(예: Service 수정 시 이를 호출하는 Controller, 혹은 API 스펙 수정 시 이를 사용하는 Front-End 컴포넌트)에서 타입이나 인자 구조가 깨지지 않는지 매핑 상태를 검증합니다.
  - **[테스트 코드]**: 해당 로직을 검증하고 있던 기존 테스트 파일(`*.test.ts`, `*.spec.ts` 등)의 검증 조건(Assertion)도 함께 수정해야 하는지 파악합니다.
- **실장 및 제어 방식**: AI는 수정을 시작하기 전, 개발자에게 "이 코드를 수정하면 `A 파일의 B 함수`와 `C 컴포넌트`에 영향이 발생하므로 함께 검토하겠습니다"라는 **영향 범위 분석 결과**를 먼저 간결하게 보고한 후 구현에 착수합니다.

### 📌 규칙 03-2. 소스코드 수정 시 인라인 주석/코멘트 정합성 검증 및 동시 업데이트

- **적용 가이드**: 기존 소스코드나 로직을 변경할 때, 오직 실행 코드(Executable Code)만 수정하고 해당 코드를 설명하던 기존의 **인라인 주석(`//`, `/** \*/`)이나 코멘트를 방치하는 행위를 전면 금지\*\*합니다. 수정된 로직의 의도와 주석의 설명이 100% 완벽히 일치하는지 반드시 검증하고 함께 수정해야 합니다.
- **품질 제어 목적**: 코드가 변경되었음에도 과거의 주석이 그대로 남아있어 발생하는 '코드와 주석의 불일치(Comment Decay)' 현상을 방지하고, 다음 개발자나 미래의 내가 코드를 오독하여 2차 버그를 생산하는 리스크를 완벽히 차단하기 위함입니다.
- **AI 필수 준수 사항**:
  - AI가 특정 함수의 내부 로직이나 조건식을 변경할 때, 해당 함수 상단에 적힌 JSDoc/TSDoc(백엔드 규칙 06 준수)이나 라인 주석의 내용을 전수 조사합니다.
  - 변경된 최신 로직에 맞게 주석의 파라미터 설명, 리턴 값 설명, 비즈니스 맥락 설명을 즉시 동기화하여 수정 코드를 제안합니다. 의미가 완전히 달라졌다면 과거의 낡은 주석은 과감히 삭제하거나 최신화합니다.

### 📌 규칙 04. 기술적 판단과 도메인 업무적 판단의 경계 준수 (Technical vs Domain Boundary)

- **적용 가이드**: AI는 코드의 구조적, 기술적 최적화에 대해서는 제안할 수 있으나, 비즈니스 핵심 로직(도메인 업무 규칙)에 대해서는 독단적으로 추측하여 실장해서는 안 됩니다.
- **권장 사항 (AI 추천 가능 영역)**:
  - 효율적인 타입 변환(Type Conversion)의 위치 및 가이드 제안 (예: 컨트롤러 vs 서비스 레이어)
  - Express 미들웨어 최적화, PostgreSQL 쿼리 튜닝, 리액트 컴포넌트 렌더링 최적화 등 기술 구조적 판단
- **전면 금지 사항 (AI 추측 실장 금지 영역)**:
  - 비즈니스 도메인 업무 처리 방식 (예: "일반적으로 정산 로직은 이렇게 처리하니 생략된 요건은 이 방식으로 구현하겠습니다" ❌)
  - 도메인 예외 처리 규칙 및 권한 관리 비즈니스 로직
- **실장 방식**: 도메인 업무 로직에 모호한 점이 있거나 명세서에 빈틈이 있을 경우, AI는 반드시 해당 로직의 구현을 비워두거나(`// TODO`) 주석으로 남긴 채 개발자에게 구체적인 도메인 규칙을 명시해 달라고 요청해야 합니다.

### 📌 규칙 05. 확인 완료 후 단일 문장 요약 및 즉각 실장 (One-Sentence Sync & Execution)

- **적용 가이드**: 개발자나 담당 프로퍼를 통해 모호한 요건, 타입 불일치, 또는 도메인 규칙에 대한 확인(Q&A)이 완료되면, AI는 즉시 코드를 짜지 않고 최종 합의된 내용을 단 한 문장으로 요약하여 제시해야 합니다.
- **AI 필수 준수 사항**:
  - **[최종 인식 맞추기]**: "확인해 주신 내용에 따라 `[최종 합의된 핵심 구현 내용]`을 반영하여 구현하겠습니다." 형태로 단 한 문장의 요약 피드백을 출력합니다.
- **실장 방식**: 요약 문장을 출력한 직후, 개발자의 추가 승인 대기 시간 없이 검증된 룰에 맞춰 해당 모듈의 소스코드 작성을 즉시 시작(즉각 구현)합니다.

### 📌 규칙 06. OpenAPI 기반 타입 자동 생성 및 구현 순서 엄격 준수 (Schema-Driven Development)

- **적용 가이드**: 프론트엔드(Next.js)와 백엔드(Express) 간의 API 통신 스펙 및 타입 불일치를 방지하기 위해, 새로운 API를 개발하거나 수정할 때는 반드시 규정된 스펙 기반 구현 순서를 선행 준수해야 합니다. 임의로 코드부터 작성하는 행위를 전면 금지합니다.
- **표준 개발 구현 순서 (3단계)**:
  1.  **[1단계: OpenAPI 문서 정의]**: 구현하고자 하는 기능의 API 엔드포인트, 요청(Request) 바디/쿼리 구조, 응답(Response) 스키마를 `OpenAPI(Swagger) 스펙 파일(YAML 또는 JSON)`로 최우선 정의합니다.
  2.  **[2단계: 타입 자동 생성 실행]**: 정의된 OpenAPI 스펙 파일을 기반으로 프로젝트의 타입 자동 생성 도구(CLI 명령어 등)를 실행하여, 프론트엔드와 백엔드가 공통으로 참조할 타입스크립트(TypeScript) 인터페이스/타입을 파일 시스템에 자동으로 갱신(Generate)합니다.
  3.  **[3단계: 자동 생성된 타입 기반 실장]**: 생성된 최신 타입을 백엔드 컨트롤러/서비스 레이어 및 프론트엔드 API 클라이언트/컴포넌트에 각각 임포트(Import)하여 실제 로직 구현에 착수합니다.
- **실장 및 제어 방식**: AI에게 API 개발을 요청할 때 반드시 "OpenAPI 스펙 정의 및 타입 자동 생성이 완료된 상태인지" 먼저 묻게 하거나, AI에게 1단계인 OpenAPI 스펙 작성부터 지시하여 타입의 싱크를 강제합니다.

### 📌 규칙 09. 동일 UI 구조 중복 출현 시 공통 컴포넌트화 선행 검토 (UI Duplication Check)

- **적용 가이드**: 단일 페이지 또는 서로 다른 화면 간에 시각적 형태(HTML 마크업 구조 및 디자인 스타일)가 완벽히 일치하거나 매우 유사한 UI 블록이 **2개 이상 중복 출현**하는 경우, 이를 인라인 코드로 중복 나열하여 구현하는 행위를 전면 금지합니다. 소스코드 작성 및 리팩토링 단계에서 반드시 이를 독립된 공통 컴포넌트(Component)로 분리할 것을 최선행하여 검토해야 합니다.
- **UI 공통화 및 검토의 목적**:
  - 향후 디자인 사양이나 레이아웃이 바뀔 때, 사방에 흩어진 중복 소스코드를 일일이 찾아 고쳐야 하는 리워크(Rework)를 차단하고 UI 단일 공급원(Single Source of Truth)을 확립하기 위함입니다.
- **AI 필수 준수 사항**:
  - AI가 화면 레이아웃을 구성하던 중 카드 목록, 반복되는 폼 섹션 등 동일 구조의 UI 패턴을 2회 이상 생성하려는 흐름을 감지하면, 즉시 작업을 중단하고 해당 UI 영역을 공통 컴포넌트로 도려낼(Extract) 설계를 구상해야 합니다.
  - 컴포넌트화 검토 시에는 앞서 정의한 프론트엔드 규칙 06번(순수 UI 담당 및 도메인 로직 배제) 및 07번(스타일 내부 완결 및 외부 className 덮어쓰기 금지)을 엄격히 계승하여 범용적인 UI 칩 형태로 설계합니다.
- **실장 및 제어 방식**: AI는 구현에 들어가기 전 개발자에게 "현재 화면 구조에서 동일한 UI 디자인 패턴이 2회 이상 발견되었습니다. 이를 `components/` 하위의 독립된 공통 컴포넌트인 `[추천컴포넌트명]`으로 먼저 분리한 뒤 조립하는 방향으로 진행할까요?"라고 제안하여 확인(승인)을 구합니다.

## 🐙 2. Git 브랜치 관리 및 충돌 해결 프로세스 (Git & Conflict Resolution)

### 📌 규칙 01. Rebase 기반 충돌 해결 및 기준 브랜치 보존

- **적용 가이드**: 작업 중 원격 저장소의 최신 변경 사항과 내 작업 본 사이에 충돌(Conflict)이 발생할 경우, 무작정 `git merge`를 실행하지 않고 **`git rebase`**를 사용하여 충돌을 해결해야 합니다.
- **Git 운영 원칙**:
  - **기준 브랜치 보존**: 팀의 중심이 되는 기준 브랜치(예: `develop` 또는 `main`)의 기존 커밋 히스토리는 원형 그대로 절대 훼손하지 않고 유지합니다.
  - **신규 브랜치 내용 취합**: 내가 작업 중인 신규 기능 브랜치(Feature Branch)의 커밋들을 기준 브랜치의 최신 커밋 뒤로 베이스를 재설정(Rebase)하며 충돌을 해결합니다.
- **실장 및 제어 방식**:
  1.  최신 기준 브랜치 내용을 로컬에 동기화합니다. (`git checkout develop` -> `git pull`)
  2.  내가 작업하던 브랜치로 이동하여 rebase를 실행합니다. (`git checkout feature/작업명` -> `git rebase develop`)
  3.  충돌이 발생하면 코드를 수동으로 확인하여 **기준 브랜치(`develop`)의 최신 코드를 안전하게 유지**한 상태에서, 내가 새로 작성한 코드를 알맞게 끼워 넣는 방식으로 취합(Merge)합니다.
  4.  충돌 해결 후 rebase 프로세스를 이어 나갑니다. (`git add .` -> `git rebase --continue`)

### 📌 규칙 02. 자동 생성 파일 충돌 시 수동 수정 금지 및 재생성 원칙

- **적용 가이드**: 빌드 도구, 프레임워크, 혹은 패키지 매니저에 의해 자동으로 만들어지는 산출물 파일(Auto-generated Files)에서 Git 충돌이 발생했을 경우, 코드를 눈으로 보며 수동으로 충돌을 수정(`Accept Incoming/Current`)하는 행위를 전면 금지합니다.
- **대상 파일 예시**:
  - 데이터베이스 관련 자동 생성 산출물 (예: ORM 클라이언트 빌드 파일, 데이터베이스 스키마 마이그레이션 자동 생성 파일 등)
  - 패키지 잠금 파일 (예: `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`)
  - 프레임워크 빌드 및 캐시 폴더 내 파일 (예: `.next/` 내부 파일)
  - 기타 컴파일러나 라이브러리가 자동 생성하는 매핑 파일
- **실장 및 제어 방식**:
  1.  충돌이 난 자동 생성 파일을 로컬에서 과감히 완전히 삭제하거나 원격 기준 브랜치의 상태로 초기화합니다.
  2.  DevContainer 환경에서 빌드 명령어 또는 설치/재생성 명령어를 다시 실행하여 충돌이 없는 깨끗한 버전으로 새롭게 파일이 빌드(재생성)되도록 유도합니다.
  3.  새롭게 재생성되어 충돌이 해결된 파일을 Git 스테이징에 추가(`git add`)하고 프로세스를 마무리합니다.

## 🔍 3. AI 소스코드 최종 제출 전 4대 영역 체크리스트 (Final Code Review Checklist)

_AI는 소스코드 작성을 완료한 후, 개발자에게 최종 결과물을 제시하기 전에 아래 4대 핵심 기준을 스스로 전수 검사하고 이상이 없음을 선언해야 합니다._

### 📂 A. 레이어의 책임 (Layered Responsibility)

- [ ] **컨트롤러 격리**: 컨트롤러가 데이터베이스 액세스 라이브러리(ORM/Query Builder)를 직접 참조하거나 DB에 직접 쿼리를 날리지 않는가?
- [ ] **서비스 순수성**: 서비스 레이어 내부에서 HTTP 요청/응답 객체(`req`, `res`)를 직접 참조하거나 HTTP 상태 코드를 직접 결정하여 반환하지 않는가? (도메인 전용 에러만 던지는가?)
- [ ] **단방향 의존성**: 의존 방향(`Route ➡️ Controller ➡️ Service ➡️ Repository`)을 역행하거나 계층을 건너뛰는 흐름이 없는가?

### 🧩 B. 타입 정합성 (Type Consistency)

- [ ] **OpenAPI 싱크**: 새로운 API나 변경 사항이 OpenAPI 스펙에 선행 정의되었고, 자동 생성된 최신 타입스크립트 인터페이스를 정직하게 임포트하여 사용했는가?
- [ ] **any 사용 금지**: 예외 상황이나 타입 불일치를 우회하기 위해 `any` 타입을 임의로 남발하지 않았는가?
- [ ] **타입 불일치 중단**: 프론트-백-DB 간의 타입 구조가 어긋난 부분이 있다면 독단적으로 실장하지 않고 개발자에게 즉시 보고하고 확인받았는가?

### 🔒 C. 시스템 보안 (Application Security)

- [ ] **하드코딩 금지**: API 키, DB 접속 문자열, 패스워드 등 민감한 자격 증명이 코드 내에 날것으로 기재되지 않고 반드시 환경 변수(`process.env`)로 처리되었는가?
- [ ] **에러 유출 금지**: 에러 응답 객체에 스택 트레이스, DB 테이블명, 내부 파일 경로 등 인프라 내부 정보가 노출되지 않도록 전역 에러 핸들러를 거쳤는가?
- [ ] **로그 마스킹**: 콘솔 및 서버 로그에 유저 개인정보(PII)나 인증 토큰이 평문으로 찍히지 않도록 마스킹 처리를 완료했는가?
- [ ] **요청 값 검증**: 외부 입력(리퀘스트 바디) 값을 무조건 맹신하지 않고, DB에서 꺼내온 내부 서비스의 반환값을 기준으로 핵심 비즈니스 연산을 수행하는가?

### ⚡ D. 애플리케이션 성능 (Performance Control)

- [ ] **N+1 문제 차단**: `for`, `while`, `map` 등 루프 처리 내에서 데이터베이스 쿼리를 실행하거나 외부 API를 호출하는 코드가 없는가? (벌크/배치 처리 완료 여부)
- [ ] **복잡도 제어**: 제어문 중첩 깊이가 최대 2단계(Depth 2)를 초과하지 않도록 조기 반환(Early Return)이나 프라이빗 함수 분할을 적용했는가?
- [ ] **클린 코드 준수**: 복잡한 조건식을 의미 있는 설명 변수로 분해하고, 추상적인 변수/함수명 대신 도메인 의도가 명확한 단어를 선택했는가?
