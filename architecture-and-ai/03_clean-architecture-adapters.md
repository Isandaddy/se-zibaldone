# architecture-and-ai/03_clean-architecture-adapters.md

# 클린 아키텍처 어댑터 패턴 및 대용량 트래픽 대응 실장 가이드

본 문서는 외부 환경(클라이언트 인터페이스, 외부 벤더 시스템 등)의 변화로부터 시스템의 핵심 비즈니스 로직을 보호하고, 대형 축구 이벤트 등 대규모 동시 접속자(부하) 발생 시 애플리케이션 소스 코드 레벨에서 안정성을 확보하기 위한 설계 지침을 정의한다.

---

## 1. 아키텍처 핵심 사상: 외부 환경으로부터의 격리

클린 아키텍처 및 헥사고날 아키텍처의 대원칙은 **"핵심 비즈니스 로직(Domain, Service)은 외부 세상(화면, CMS, 티켓 사이트, 벤더)이 어떻게 변경되든 절대 영향을 받지 않아야 한다"**는 것이다. 이를 위해 진입점과 진출점에 **어댑터(Adapter) 패턴**을 철저히 적용한다.

---

## 2. 인바운드 어댑터 (Inbound Adapter): 복수 진입점 대응

본 프로젝트의 세트 티켓 처리는 외부 CMS(관리자용)와 티켓 사이트(일반 사용자용) 등 접근하는 진입점이 복수로 존재하며, 각 클라이언트가 요구하는 데이터 포맷과 트래픽 특성이 완전히 다르다.

### ① 설계 규칙

- **컨트롤러 및 DTO 분리:** 외부 CMS용 API 컨트롤러와 일반 티켓 사이트용 API 컨트롤러를 물리적으로 명확히 분리한다.
- **진입점별 전용 DTO 사용:** 각 진입점에 맞는 전용 DTO(Data Transfer Object)를 두고, 내부 서비스로 넘어가기 전 어댑터(입구) 단계에서 핵심 도메인 모델로 변환(Mapping)하여 전달한다.

### ② 대규모 부하 상황에서의 이점

- 대형 축구 이벤트로 인해 일반 티켓 사이트 측 진입점(`TicketSiteController`)에만 트래픽이 수만 배 폭증해도, 비즈니스 핵심부와 CMS 컨트롤러 영역은 안전하게 격리된다.
- 나중에 특정 클라이언트의 UI가 개편되어 화면 스펙(DTO)이 바뀌더라도, 해당 어댑터 내부의 매핑 로직 한 줄만 수정하면 되며 핵심 비즈니스 서비스(`Service`)는 단 한 줄도 변경할 필요가 없다.

---

## 3. 아웃바운드 어댑터 (Outbound Adapter): 외부 시스템 종속성 제거

벤더 솔루션(오케스트레이터)이나 외부 마이크로서비스가 제공하는 데이터 모델 규격(껍데기)이 우리 시스템의 심장부(Service, Entity)까지 깊숙이 침투하는 것을 차단한다.

- **대책:** 외부 시스템의 응답을 받는 출구(인터페이스/어댑터) 단계에서 즉시 우리 시스템 전용의 깔끔한 도메인 모델로 번역(Mapping)하는 구조를 취한다. 이를 통해 외부 시스템의 필드명이 변경되거나 시스템이 교체되더라도 우리 내부 코드가 깨지는 연쇄 작용을 방지한다.

---

## 4. 애플리케이션 레벨의 대용량 부하 분산 실장 지침 (AI 코딩 지시용)

인프라 아키텍처 환경 외에, AI 에이전트(Claude Code)에게 설계를 시키고 실제 코드를 짜게 할 때 반드시 포함해야 할 코드 레벨의 성능 최적화 지침이다.

### ① 외부 시스템 조회 결과의 '로컬 캐싱(Caching)'

- **지침:** 변하지 않는 외부 시스템의 마스터 데이터(일정 정보, 클럽 정보 등)는 매 요청마다 외부 API를 호출하지 않고, 애플리케이션 메모리 영역(`Spring Cache`, `Node-Cache` 등)에 캐싱하여 즉시 반환하도록 코드를 구현한다.

### ② 읽기(조회)와 쓰기(등록)의 분리 및 동시성 제어

- **지침:** 대량의 조회 요청이 DB 병목을 일으키지 않도록 구조를 분리한다. 더불어 한정된 재고를 가진 세트 티켓에 수많은 구매/등록 요청이 동시에 몰릴 것에 대비하여, 데이터베이스의 **낙관적 락(Optimistic Lock)** 또는 **비관적 락(Pessimistic Lock)**을 활용한 동시성 제어 로직을 필수적으로 구현한다.

### ③ 논블로킹(Non-blocking) / 비동기 통신 구조

- **지침:** 타 시스템 연동 시 외부 서버의 응답을 기다리느라 우리 시스템의 스레드(Thread)가 고갈되는 블로킹 방식을 지양한다. 스레드를 차단하지 않는 비동기/논블로킹 클라이언트(`Spring WebClient` 등)를 사용하여 동시 접속 대응 능력을 극대화한다.
