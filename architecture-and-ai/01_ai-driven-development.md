# AI 주도 개발 프레임워크 (Specification-Driven Development)

본 문서는 프로젝트 개발 시 AI 코딩 에이전트(Claude Code, GitHub Copilot 등)와 협업하여 안전하고 신속하게 시스템을 구축하기 위한 **SDD(Specification-Driven Development, 스펙 중심 개발)** 프로세스 및 규칙을 정의한다.

---

## 1. 개요: Spec Kit과 `cc-sdd` 세대의 차이

기존의 Spec Kit이나 디자인 가이드라인이 **'인간 디자이너와 인간 개발자 간의 소통'**을 위한 문서였다면, `cc-sdd` 프레임워크는 **'AI 에이전트가 인간의 비즈니스 의도를 오해 없이 이해하고 스스로 코딩할 수 있도록 최적화된 명세(Specification)'** 작성을 목표로 한다.

- **사상:** 코드를 먼저 짜지 않는다. 완벽한 구조화 마크다운(Markdown) 명세를 빌드하고 승인한 뒤 구현을 시작한다.
- **이점:** 개발자는 복잡한 구현(실장) 코드 작성 스트레스에서 벗어나 아키텍처 검토 및 비즈니스 검증(Reviewer) 역할에 집중할 수 있다.

---

## 2. 핵심 용어 정의 (Kiro & Spec)

### ① Kiro (키로)

AI 자율 워크플로우를 구동하기 위한 엔진 및 명령어 프레임워크의 접두사(Prefix)이다. AI에게 작업을 탐색하고 실장하도록 지시하는 통제 명령 세트를 의미한다.

- 프로젝트 루트에 `.kiro/` 폴더를 생성하여 모든 AI용 명세서와 규칙을 관리한다.

### ② Spec (스펙 / 명세)

단순한 기능 정의 요약본이 아니다. AI가 비즈니스 로직을 한 줄도 틀리지 않고 구현할 수 있도록 **[요구사항 정의 $\rightarrow$ 외부/내부 아키텍처 설계 $\rightarrow$ 세부 구현 태스크]** 단계를 거쳐 체계적으로 구조화된 마크다운(.md) 문서이다.

---

## 3. 인간과 AI의 협업 워크플로우 (3단계)

### [1단계] 탐색 및 초안 작성 (`/kiro-discovery`)

- **주체:** AI 에이전트
- **행동:** 인간 개발자가 제공한 원시 요구사항(예: 엑셀 요건정의서, 러프한 아이디어 기획안)을 바탕으로 프로젝트의 기술 스택과 MSA 맥락을 분석하여 `.kiro/specs/` 하위에 구조화된 마크다운 스펙 문서 초안을 자동으로 생성한다.

### [2단계] 인간 시니어의 리뷰 및 승인 (`Human Review`)

- **주체:** 인간 개발자 (본인)
- **행동:** AI가 작성한 마크다운 설계도를 검토한다. 동시성 이슈, 분산 트랜잭션 예외 처리, 타 시스템 연동 타임아웃 등 시니어 레벨에서 고려해야 할 아키텍처적 허점이 있다면 마크다운을 직접 수정하거나 AI에게 수정을 재요청한다. 설계가 완벽하다고 판단되면 명세를 **'승인(Approved)'**한다.

### [3단계] 자율 구현 및 검증 (`/kiro-impl`)

- **주체:** AI 에이전트 (Claude Code)
- **행동:** 승인된 마크다운 스펙 내의 `## 구현 세부 태스크 (Tasks)`를 파싱하여, 인간의 개입 없이 스스로 파일 구조를 생성하고 코드를 작성한다. 코드 구현 직후 자체적으로 컴파일 및 테스트 코드를 실행하여 자율적으로 디버깅을 마친 후 최종 결과물만 인간에게 보고한다.

---

## 4. AI에게 코딩을 시키기 위한 마크다운 명세(Spec) 표준 템플릿

AI에게 명세서 초안을 작성하라고 지시할 때, 반드시 아래 구조를 유지하도록 가이드한다.

```markdown
# Spec: [기능 명칭 및 타겟 서비스명]

## 1. 배경 및 비즈니스 요건 (Requirements)

- [ ] 인간의 요구사항을 명확한 체크리스트 형태로 세분화

## 2. 외부 시스템 연관성 및 인터페이스 (External Interfaces)

- 연동할 마이크로서비스(MSA)명 및 프로토콜 규격 (REST/gRPC/Event)
- 호출 실패 시의 에러 핸들링 및 타임아웃 규격 명시

## 3. 내부 시스템 설계 (Internal Design)

- 데이터 모델 (DTO 및 Entity 구조)
- 클린 아키텍처 준수 여부 (Inbound/Outbound Adapter 배치)

## 4. 구현 세부 태스크 (Implementation Tasks)

- [ ] Task 1: [구체적인 파일 생성 및 컴포넌트 단위 구현 작업]
- [ ] Task 2: [Mocking을 활용한 타 시스템 연동부 및 예외 처리 구현]
- [ ] Task 3: [도메인 비즈니스 로직 및 통합 테스트 코드 구현]
```
